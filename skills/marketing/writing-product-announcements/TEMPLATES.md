# Templates

## Default announcement structure
Use this as the base for all announcement types:

```md
## [Headline: Benefit or change statement]

[One sentence: What changed and when it's available.]

### What's new
[2-4 sentences describing the change. Focus on user impact.]

### Why it matters
[1-2 sentences on benefit or reason for change.]

### What to do
[Specific next step. Link to docs, settings, or action.]

[CTA button/link text]
```

## Launch announcement (new feature)

```md
## Introducing [Feature Name]: [Benefit statement]

[Feature Name] is now available [to all users / in beta / for Pro plans].

### What you can do
- [Capability 1]
- [Capability 2]
- [Capability 3]

### Get started
[One sentence with link to docs or feature.]

[Try it now →]
```

## Update announcement (improvement)

```md
## [Feature Name] just got better

We've improved [Feature Name] to [specific improvement].

### What changed
[Bullet list of changes]

### What this means for you
[1-2 sentences on benefit]

No action required—these improvements are already live.
```

## Deprecation announcement

```md
## [Feature Name] is being retired on [Date]

Starting [Date], [Feature Name] will no longer be available.

### Why we're making this change
[1-2 sentences explaining reason]

### What you need to do
1. [Action 1 with deadline]
2. [Action 2 if applicable]

### Recommended alternative
[Name and link to replacement, if any]

Questions? [Contact support / See FAQ]
```

## Breaking change announcement

```md
## Action required: [Change] on [Date]

On [Date], we're changing [specific behavior]. This may affect [who/what].

### What's changing
[Clear description of old → new behavior]

### Who's affected
[Specific criteria]

### Required action by [Date]
1. [Step 1]
2. [Step 2]

### Need help?
[Link to migration guide or support]
```

## Changelog entry (minimal)

```md
### [Date] — [Category]

**[Feature/Fix name]**: [One sentence description.] [Learn more →]
```
